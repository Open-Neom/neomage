# Architecture Decision Framework
Requirements drive architecture. Trade-offs inform decisions. ADRs capture rationale.
- Start simple; add complexity ONLY when proven necessary -- removing complexity is harder than adding it
- Every decision needs explicit trade-off analysis: what you gain vs. what you lose
- Classify projects by scale (MVP/SaaS/Enterprise) to right-size architectural patterns
- Use decision trees for pattern selection; match patterns to actual constraints, not hype
- Validate that requirements are clearly understood before choosing any architecture
- Identify and document all constraints (team size, budget, timeline, compliance)
- Always consider simpler alternatives before adopting complex patterns
- Ensure team expertise matches chosen patterns -- unfamiliar patterns cause more harm than benefit
- Write Architecture Decision Records (ADRs) for every significant decision
- Simplicity is the ultimate sophistication: you can always add patterns later
